# 银行业UAT测试案例编写规范

## 1. 编写原则
- 可执行性：测试步骤清晰，业务人员可以直接按照步骤执行测试
- 覆盖性：覆盖所有需求点、业务规则、异常场景
- 独立性：每个测试案例独立，不依赖其他测试案例的执行结果
- 可验证性：预期结果明确，可量化、可检查

## 2. 优先级定义
- 高：核心业务功能、涉及资金交易、监管合规要求的功能
- 中：辅助业务功能、用户体验相关功能
- 低：边缘功能、展示类功能

## 3. 场景分类要求
### 3.1 正常流程场景
覆盖需求描述的主流程、正常操作路径下的功能验证

### 3.2 异常流程场景
- 输入非法值、空值、超限值的场景
- 业务规则不满足时的拦截场景
- 系统异常、网络中断等异常场景
- 权限不足时的操作场景

### 3.3 边界场景
- 数值边界（最大/最小金额、最大/最小笔数等）
- 时间边界（日切时间、节假日、业务截止时间等）
- 权限边界（不同角色操作权限边界）

## 4. 编写注意事项
- 避免使用技术术语，使用业务人员易懂的业务语言
- 测试步骤描述要具体，避免模糊表述（如"正常操作"改为"输入正确的账号密码点击登录"）
- 预期结果要明确，包含界面提示、数据变化、业务状态变化等所有可验证的内容
- 涉及金额的场景需要明确金额精度、计算规则
- 涉及多角色的场景需要明确操作角色
